构建安全服务

Spring Security

SpringSecurity是一个专注于为Java应用程序提供身份认证和授权的框架，它的强大之处在于他可以轻松拓展以满足自定义的需求。

特征

  • 对身份的认证和授权提供全面的、可拓展的支持
    • 认证登录，只有登录用户才能看到的内容
    • 授权，即使是登录的用户，也有不同的权限，对不同角色进行不同的授权
  • 防止各种攻击，如会话固定攻击、点击劫持、csrf攻击等
  • 支持与Servlet API、Spring MVC等Web技术集成

CSRF原理

某一网站盗取Cookie中的凭证（ticket），从而模拟用户提交表单。

Security解决方案：引入Security后，用户请求提交表单时，Security会在表单中加一个隐藏的Token，其他网站能够窃取到你的Ticket，但是无法获得你的表单数据（其中的Token），从而会被服务器发现。

Redis高级数据类型

  • HyperLogLog：超级日志
    • 采用一种基数算法用于完成独立总数的统计
    • 数据占用空间少，无论统计多少数据，只占12K的内存空间
    • 不精确的统计算法，标准误差为0.81%
  • Bitmap：位图
    • 不是一种独立的数据结构，实际上就是字符串
    • 支持按位存取数据，可以将其看成是byte数组
    • 大量存储连续数据的布尔值，如每天的签到。如一年的签到数据：365/8，只有40多个字节

两个数据类型都适合用来给网站运营的数据进行统计，并且节约内存

网站数据统计

  • UV(Unique Visitor)

    • 独立访客，通过IP地址去重统计，希望把匿名用户（游客）也统计起来
    • 每次访问都要统计，因为不确定此次访问的IP是否已经记录过
    • 用HyperLogLog进行统计，性能好，且存储空间小。

  • DAU（Daily Active User）

    • 日活跃用户
    • 访问过一次则认为其是活跃用户。
    • 与UV的区别是：通过userId进行去重（已注册用户），而且是比较精确的结果，故使用bitmap。可以用userId为Index，在redis中存储，key为日期，所以每天需要大概几十万位。

部署在服务器上运行的程序有两种，一种是通过请求访问执行的，另一种是定时任务执行的。

在分布式部署中，我们可以通过nginx做到负载匀衡的去控制每个请求在不同服务器上的访问。但是定时任务不需要ngnix就可以自动的在不同的服务器上访问。如何对同一定时任务，在不同服务器上做到协同是关键。

  • Spring和jdk的定时任务是基于内存的，他们的配置参数是基于内存的，分布式部署时，各台机器内存不共享，所以无法做到协同定时执行。

  • Quartz是基于数据库的，数据库是放在一台单独的服务器上，只有一个。

    使用Quartz，不同服务器的Scheduler运行时，会在数据库服务器访问运行数据，通过排队（加锁），可以使得程序只运行一次。互斥访问做到协同

优化网站性能

加缓存，并使用压力测试工具，展示性能变化情况。

  • 本地缓存
    • 数据存到应用服务器上
    • 常用工具：Ehchche，Guava，Caffeine
  • 分布式缓存
    • 缓存数据存到NoSQL服务器上（有一些数据必须放在分布式服务器上。跨服务器）
    • 比本地缓存差（有网络开销）
    • 常用缓存工具：MemCache、Redis
  • 多级缓存
    • 一级缓存（本地）---->二级缓存（分布式缓存） ----> DB
    • 总之尽力避免缓存雪崩，尽力避免访问DataBase

目标：优化热门帖子列表（对它缓存）（按时间帖子列表经常变，所以要经常更新缓存，性能下降）

本地缓存：Caffeine（可以用Spring整合，但是不建议，因为Spring只用一个缓存管理器来管理所有缓存，所以缓存的大小/过期时间都是一样的，不便于自定义）
